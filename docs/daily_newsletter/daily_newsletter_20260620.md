## AI

### 1. 100体のAIを1人で操る——AI時代の認知負荷をどう減らすか
**Zenn / ShintaroAmaike**
[https://zenn.dev/shintaroamaike/articles/dfe5f0aac80c16](https://zenn.dev/shintaroamaike/articles/dfe5f0aac80c16)

複数のAIエージェントを並行稼働させることで生産性は向上した一方、「確認・判断・次の指示」というループが増殖し、人間の認知負荷が限界に近づいている——という問題提起から始まる記事。著者は5つのUIアーキテクチャを提案している。

1. **空間キャンバス型UI**：変更をノードとして配置し、依存関係を線で結ぶ。時系列ではなく位置関係でプロジェクト全体を把握する。
2. **シグナル中心型UI**：AIが「低リスク」と判断したものは表示を消し、本当に注意すべきシグナル（AI同士の見解の相違・過去のバグパターン一致・ビジネス影響）だけを残す。
3. **バックグラウンド型UI**：AIは裏側で動き、「Go/No-Go判断が必要」「セキュリティリスクを検出」など本当に必要な瞬間だけ割り込む。
4. **シミュレーション型UI**：変更を適用する前に仮想環境で試し、パフォーマンス・コスト・エラー率の変化をグラフで予測する。
5. **パーソナルAIオーケストラ・ダッシュボード**：価値観と判断パターンを学習したメタAIが全エージェントを束ね、1日の終わりに振り返りを提示する。

共通する設計思想は「人間に流す情報量を減らす」こと。チャットUIのままでは人間が「AIの成果物をApproveし続ける係」になるという指摘は、AIプロダクト設計の本質を突いている。

### 3. Strands Agentsでツールを実行するAIエージェントを2つ実装した記録
**Zenn / satoru_o**
[https://zenn.dev/satoru_o/articles/efaa935d1e4cdd](https://zenn.dev/satoru_o/articles/efaa935d1e4cdd)

AWSのオープンソースSDK「Strands Agents」を使い、MCPサーバー経由のファイル操作エージェントとgit diffからコミットメッセージを提案するエージェントを実装した実録記事。

### 7. GitHub Copilotのクレジット消費がユーザー単位でAPI取得可能に：監視を自動化する ai_credits_used の使い方
**Zenn / ヘッドウォータース（こばやし うきょう）**
[https://zenn.dev/headwaters/articles/github-copilot-ai-credits-usage-metrics-2026](https://zenn.dev/headwaters/articles/github-copilot-ai-credits-usage-metrics-2026)

2026年6月19日、GitHub Copilot usage metrics APIに`ai_credits_used`フィールドが追加され、ユーザー単位のクレジット消費をプログラムから取得できるようになった。これにより、重利用ユーザーの特定・消費の偏りの可視化・ユーザー予算設定の根拠データ取得が自動化できる。

ただし、以下の4つの落とし穴に注意が必要。

| 急所 | 要点 |
|---|---|
| ① 内訳は取れない | feature/model/surface別ではなくユーザー単位の総計のみ |
| ② 請求額ではない | 分析用メトリクス。確定請求額はbilling側が正 |
| ③ ラグがある | データ反映は最大2営業日後。リアルタイム監視は不可 |
| ④ 権限が要る | エンタープライズ管理者・組織オーナー＋ポリシー有効化が前提 |

「予算制御（ハードストップ）」と「観測（トレンド把握）」の両輪を揃えることで初めて従量課金の運用が回るという整理は、AI基盤の運用設計として参考になる。

### 8. AIエージェントが自律的に自己改善する self-improvement スキルが良い
**Zenn / ｓｔａｋ**
[https://zenn.dev/stak22/articles/3ea03d0e3ca025](https://zenn.dev/stak22/articles/3ea03d0e3ca025)

Claude Codeを使った開発で「同じミスを繰り返す」問題を解決する`self-improvement` Agent Skillの紹介記事（6/17公開）。

このスキルは、エージェントが会話の中で生まれた「学び」を構造化して`.learnings/`配下のMarkdownファイルに自動で蓄積する仕組み。訂正・feature request・knowledge gap・best practice・エラー解決の5種類のトリガーを検知し、ID・日時・要約・詳細・対応案を記録する。

特に優れているのは、学びの保存先がコードと同じリポジトリであるため、**コミット一発でチーム全員のエージェントに学びが共有される**点。また、同じ学びが3回以上記録されると`CLAUDE.md`や`AGENTS.md`への昇格を促す二段構えの設計により、ノイズに埋もれず重要なルールを確立できる。
