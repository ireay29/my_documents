## AI

### 1. 時雨堂のスキル設計、全容を公開
**voluntas.ghost.io** | [記事を読む](https://voluntas.ghost.io/shiguredo-skill-20206-06/)

WebRTCスタックで知られる時雨堂が、自社のAIエージェント運用で使用している「スキル」の体系を公開した。`shiguredo-git`（コミット規約）、`polish-issue`（issueをLLMで磨き上げる）、`review-diff-code`（差分レビューを最低3周）など、開発フロー全体をスキルで覆う設計が特徴的。特に、スキルをインストールしていないメンバーが「LLMがあり得ないくらいバカになった」と感じるほど効果があったという証言は、SKILL.mdの重要性を端的に示している。**人間の仕事は「判断」に集約されつつある**という現場の声が印象的。

### 2. Claude Code vs Codex — 設計思想の違いを解剖
**Zenn** | [記事を読む](https://zenn.dev/tark_ann/articles/e8b09c6db73bfb)

Claude CodeとCodexは同じ「AIコーディングエージェント」でありながら、設計思想が根本的に異なる。Claude Codeは「対話しながら進める」インタラクティブ志向、Codexは「タスクを渡して待つ」非同期志向。どちらが優れているかではなく、**タスクの性質によって使い分ける**ことが重要という視点は、AIツールの選定基準として実践的な示唆を与えてくれる。

### 3. AIによる半自動PRレビューの実装
**Zenn** | [記事を読む](https://zenn.dev/lingmu/articles/2026-06-21-semi-auto-pr-review)

GitHub ActionsとClaude APIを組み合わせ、PRのdiffを自動でレビューする仕組みを構築した事例。完全自動化ではなく「半自動」にとどめた理由として、**最終判断は人間が行うべき**という設計思想が語られている。レビューの観点をプロンプトで制御することで、チームのコーディング規約に沿ったフィードバックを安定して得られるようになったという。

### 4. Claude Codeのpush事故を防ぐ設定
**Zenn** | [記事を読む](https://zenn.dev/syunpp/articles/813903a959d155)

Claude Codeが意図せずリモートリポジトリにpushしてしまう事故を防ぐための設定方法を解説。`CLAUDE.md`に「pushは必ず人間の確認を得てから」と明記することで、エージェントの行動範囲を制御できる。**AIに渡す権限の境界をどこに引くか**という問いに対する、シンプルかつ実践的なアプローチ。

### 5. AI以後の受託開発 — 変わること、変わらないこと
**Zenn** | [記事を読む](https://zenn.dev/tark_ann/articles/e8b09c6db73bfb)

AIエージェントが普及した後の受託開発の在り方を論じた記事。コードを書くコストが下がる一方で、「何を作るかを決める」「顧客の課題を言語化する」という上流工程の価値はむしろ高まるという主張。**AIが得意なことと人間が得意なことの再定義**が、今後のエンジニアキャリアの核心になるという視点は説得力がある。

## 心理学

### 6. ものごとの移ろいに想いを寄せる、夏至のころ
**大嵜直人のブログ（はてなブログ）** | [記事を読む](https://oosakinaoto.com/entry/when-we-reflect-on-change-around-the-summer-solstice)

本日は夏至。文筆家・心理カウンセラーの大嵜直人氏が、七十二候「乃東枯（なつかれくさかるる）」に寄せて、変化と移ろいについて書いている。「絶好調と感じているときほど、実は山を下りているのかもしれない。もうどうしようもないと絶望しているときほど、光の種は落ちている」という一節は、アドラー心理学の「勇気づけ」にも通じる、変化を恐れずに受け入れる姿勢を示している。AIの急速な進化に揺れる今だからこそ、季節のめぐりが教えてくれる「変わらないものなど何もない」という視点は、心の安定剤になりうる。

## ビジネス

### 7. 自治体がVibe Codingで「書かない窓口」を内製し、OSSとして公開
**note（北海道芽室町）** | [記事を読む](https://note.com/memuro_dx_oss/n/nf982351d30ef)

北海道芽室町の窓口職員が、専門のエンジニアなしにAIを使って行政窓口システムを内製し、**MADO（Memuro Agile Desk Open）** としてOSS公開した事例。住民が同じ情報を55回書くという課題を、申請書の自動転記で解消した。「捨てるつもりで作った試作が本番稼働した」という経緯も含め、Vibe Codingの民主化を象徴する事例として注目度が高い。売らずにOSSを選んだ理由として「属人化を解消するために作ったものを、売ることで新しい属人化に変えてしまう」という論理は鋭い。

AIエージェントが開発の主役になりつつある今、「SOLID原則」「DRY原則」「テスト駆動開発」といった従来のソフトウェア工学の原則がどう変わるかを論じた記事。エージェントはコードを大量生成するため、**可読性よりも意図の明確さ**がより重要になるという主張は興味深い。人間がレビューするコードとエージェントが読むコードでは、最適な書き方が異なる可能性を示唆している。

### 10. AIデバッガーが抱える根本的な問題
**Zenn** | [記事を読む](https://zenn.dev/mattyatea/articles/6b7e6c0d6ef3e7)

AIによる自動デバッグツールの限界を論じた記事。AIは「エラーメッセージに対して回答を生成する」ことは得意だが、「なぜそのエラーが起きているのかを文脈から推論する」ことは依然として苦手という指摘。特に、**複数のシステムにまたがる因果関係の追跡**はAIが最も苦手とする領域であり、デバッグにおける人間の役割はまだ大きいという結論は、現場感覚に即している。
