## エグゼクティブサマリー

本レポートは、PetSense アプリの設計ドキュメント（`PetSense_App_Design_Document.md`）と、実際のリポジトリ（`co2-taichi/PetSense`）に含まれるモバイルアプリ（React Native / Expo）およびバックエンドサーバー（Express + Prisma + PostgreSQL）のコードを照合し、仕様上の差異と未実装項目を体系的に整理したものです。

分析の結果、**合計 47 件のギャップ**を特定しました。内訳は、**Critical（重大）が 8 件**、**High（高）が 14 件**、**Medium（中）が 15 件**、**Low（低）が 10 件**です。特に、Bluetooth デバイス通信の完全欠如、テストコードの不在、セキュリティ対策の未実装、および通知システムの未構築は、MVP リリースに向けて最優先で対処すべき課題です。

一方で、認証フロー（ログイン・登録・トークンリフレッシュ）、ローカル/リモートのデータソース切り替え機構、健康アラートのロジック層、およびレポート画面の基本構造など、設計ドキュメントの範囲を超えた先行実装も確認されました。

## データモデルのギャップ

データモデルは、設計ドキュメントと実装の間で最も差異が大きい領域です。以下に、モデルごとの詳細な比較を示します。

### User モデル

設計ドキュメントでは、ユーザーモデルに `profile_image_url`、`phone`、`address`、`is_active` といったフィールドが定義されていますが、実装ではこれらのフィールドがすべて欠落しています。

| フィールド | 設計ドキュメント | Prisma（サーバー） | SQLite（アプリ） | 状態 |
| :--- | :---: | :---: | :---: | :--- |
| `id` | あり | あり | あり | 一致 |
| `email` | あり | あり | あり | 一致 |
| `password_hash` | あり | あり（`passwordHash`） | なし | サーバーのみ |
| `name` | あり | あり | あり | 一致 |
| `profile_image_url` | あり | **なし** | **なし** | **未実装** |
| `phone` | あり | **なし** | **なし** | **未実装** |
| `address` | あり | **なし** | **なし** | **未実装** |
| `is_active` | あり | **なし** | **なし** | **未実装** |
| `refreshTokens` | なし | あり | なし | 設計外の追加 |

### Pet モデル

ペットモデルは、設計と実装で構造的なアプローチが異なります。設計では `owner_id` による直接的な所有者紐付けを想定していますが、実装では `PetUser` 中間テーブルを介した多対多の関係に変更されています。この変更自体は家族共有機能を考慮した合理的な判断ですが、設計ドキュメントとの乖離として記録する必要があります。

| フィールド | 設計ドキュメント | 実装 | 状態 |
| :--- | :---: | :---: | :--- |
| `type` / `species` | `'dog' \| 'cat' \| 'other'` | `'cat' \| 'dog'` | `'other'` 未対応 |
| `color`（毛色） | あり | **なし** | **未実装** |
| `microchip_id` | あり | **なし** | **未実装** |
| `profile_image_url` | あり | **なし**（`icon` 絵文字で代替） | **未実装** |
| `medical_history` | あり | **なし** | **未実装** |
| `targetWeightKg` | なし | あり | 設計外の追加 |
| `activityLevel` | なし | あり | 設計外の追加 |
| `toiletStyle` | なし | あり | 設計外の追加 |
| `conditions` | なし | あり | 設計外の追加 |

> **注記**: 実装側で追加された `targetWeightKg`、`activityLevel`、`toiletStyle`、`conditions` は、犬猫兼用プラットフォームとしての差別化要素であり、設計ドキュメントへの反映が推奨されます。

### Measurement モデル

計測記録モデルは、設計と実装で根本的なアーキテクチャが異なります。設計ドキュメントでは `measurement_type`（`'urine' | 'feces' | 'weight'`）による計測タイプの分離を想定していますが、実装では 1 レコードに体重・尿・便のすべてのデータを統合する「フラット構造」を採用しています。

| 項目 | 設計ドキュメント | 実装 | 影響 |
| :--- | :--- | :--- | :--- |
| **構造** | タイプ別分離 | 統合型（1レコード = 全項目） | データ粒度の低下 |
| `device_id` | あり（自動計測元の追跡） | **なし** | デバイス起源の追跡不可 |
| `measurement_type` | あり | **なし** | タイプ別フィルタリング不可 |
| 尿タンパク質 | `'negative' \| 'trace' \| 'positive'` | `'negative' \| 'positive'` | `'trace'`（微量）が欠落 |
| 尿ケトン体 | 同上 | 同上 | 同上 |
| 白血球 | `urine_wbc` | `urineLeukocytes` | 名称変更のみ |
| 赤血球 | `urine_rbc` | `urineBlood` | 名称変更のみ |

この統合型アプローチは、日次の健康スナップショットとしては合理的ですが、個別の排泄イベント（例: 1 日に複数回の排尿）を正確に記録する場合には不十分です。特に、IoT デバイスからの自動計測データを取り込む際には、イベント単位のレコードが必要になるため、将来的な再設計が必要です。

### Meal モデル

| フィールド | 設計ドキュメント | 実装 | 状態 |
| :--- | :---: | :---: | :--- |
| `meal_type` | あり（`'breakfast' \| 'lunch' \| 'dinner' \| 'snack'`） | **なし** | **未実装** |
| `food_name` | あり | `label`（名称変更） | 実質一致 |
| `calories` | あり | **なし** | **未実装** |
| `notes` | あり | **なし** | **未実装** |
| `fed_at` | あり（給餌日時） | `time`（時刻文字列のみ） | 日付情報の欠落 |
| `hydrationMl` | なし | あり | 設計外の追加 |

### Note モデル

設計ドキュメントではメモに `title`、`content`、`image_url`、`tags[]` を持つリッチなモデルを想定していますが、実装では `body`（本文テキスト）のみの最小構造となっています。画像添付やタグ付けによる分類機能は完全に欠落しています。

### Alert モデル

設計ドキュメントではアラートをデータベースに永続化し、`alert_type`、`severity`、`measurement_id`（関連計測記録）、`is_read`（既読フラグ）を持つモデルを定義しています。一方、実装では `health.ts` の `buildAlerts()` 関数が計測データからアラートを動的に生成する方式を採用しており、データベースへの永続化は行われていません。

この方式の利点は、常に最新のデータに基づいてアラートが再計算される点ですが、以下の制約があります。

- アラート履歴の追跡が不可能（過去のアラートを遡れない）
- 特定の計測記録とアラートの紐付けができない
- 既読/未読の状態がアプリ再起動時にリセットされる（ローカル state のみ）
- プッシュ通知との連携ができない

### Device モデル

| フィールド | 設計ドキュメント | 実装 | 状態 |
| :--- | :---: | :---: | :--- |
| `model` | あり | **なし** | **未実装** |
| `serial_number` | あり | **なし** | **未実装** |
| `mac_address` | あり | **なし** | **未実装** |
| `type` | `'scale' \| 'urine_sensor' \| 'camera' \| 'feeder' \| 'activity_tracker'` | `'smart_sheet' \| 'smart_base' \| 'feeder' \| 'activity_collar' \| 'water_sensor'` | 定義不一致 |
| `is_primary`（PetDevice） | あり | **なし** | **未実装** |

デバイスタイプの定義が設計と実装で異なっている点は、ハードウェア仕様の進化を反映した変更と考えられますが、設計ドキュメントの更新が必要です。

## 画面・UIのギャップ

### ナビゲーション構造

設計ドキュメントでは 5 タブ構成（ホーム、記録、食事、アラート、設定）を定義していますが、実装では 6 タブ構成となっており、**レポート**タブが追加されています。また、設計では React Navigation を使用したスタックナビゲーションを想定していますが、実装では `App.tsx` 内のカスタムタブシェルで画面を切り替える方式を採用しています。

| 画面 | 設計ドキュメント | 実装状態 | 備考 |
| :--- | :---: | :---: | :--- |
| ホーム | あり | 実装済み | 健康ダッシュボード、クイックアクション |
| 記録 | あり | 部分実装 | 週表示のみ、グラフなし |
| 食事 | あり | 部分実装 | カロリー表示なし |
| アラート | あり | 部分実装 | 永続化なし、詳細画面なし |
| レポート | **なし** | 実装済み | 設計外の追加 |
| 設定 | あり | 部分実装 | サブ画面が簡素 |
| ログイン | なし（暗黙的） | 実装済み | ゲストモード対応 |

### 未実装の画面・機能

以下の画面および機能は、設計ドキュメントで定義されているにもかかわらず、実装されていません。

**記録画面（RecordsScreen）に関する未実装項目**として、計測タイプ選択（尿・フン・体重の個別入力）が挙げられます。現在の実装では、すべての計測値を 1 つのモーダルで同時に入力する方式となっています。また、グラフ表示（1 週間・1 ヶ月・3 ヶ月の切り替え）は UI 上にトグルが存在するものの、実際には週表示のみが機能しており、月次・四半期表示は未実装です。さらに、計測詳細画面への遷移、および記録の編集・削除機能も欠落しています。

**食事画面（MealsScreen）に関する未実装項目**として、食事タイプの選択（朝食・昼食・夕食・おやつ）、カロリー入力・表示、食事詳細画面、および給餌日時の編集が挙げられます。現在の実装では、ラベルとグラム数のみの簡素な入力フォームとなっています。

**アラート画面（AlertsScreen）に関する未実装項目**として、アラート詳細画面、獣医師への相談ボタン、およびアラートルールのカスタマイズ（閾値設定）が挙げられます。現在の実装では、ローカルの通知トグルとペアリングトグルが配置されていますが、実際の通知送信ロジックには接続されていません。

**設定画面（SettingsScreen）に関する未実装項目**として、ユーザープロフィール編集（画像、電話番号、住所）、ペット編集・削除機能、デバイスの Bluetooth ペアリングフロー、通知設定の詳細（閾値、頻度、通知先）、およびアプリ設定（言語、テーマ、データエクスポート）が挙げられます。

**ペット登録フォームの不足フィールド**として、毛色（`color`）、マイクロチップ ID（`microchip_id`）、プロフィール画像（`profile_image_url`）、および医療履歴（`medical_history`）の入力欄が欠落しています。犬種・猫種の入力は自由テキストのみで、マスターデータに基づく選択肢の提示がありません。

## Bluetooth・デバイス通信のギャップ

設計ドキュメントでは、`react-native-ble-plx` を使用した Bluetooth Low Energy（BLE）通信を定義しており、デバイス追加フローとして「デバイスタイプ選択 → Bluetooth ペアリング → デバイス名設定 → ペットへの紐付け」という 4 ステップのフローを想定しています。

**実装の現状は、BLE 関連のコードが一切存在しません。** デバイスの「接続」は、`updateDeviceConnection()` 関数による `paired` / `offline` の文字列切り替えのみで実現されており、実際の Bluetooth 通信は行われていません。デバイス登録も、名前とファームウェアバージョンを手動入力するだけの簡素なフォームです。

PetSense の中核的な価値提案である「IoT デバイスによる自動計測」を実現するためには、以下の仕様策定と実装が必要です。

- BLE スキャン・ペアリングプロトコルの定義（GATT サービス UUID、キャラクタリスティック定義）
- デバイスファームウェアとの通信プロトコル（データフォーマット、コマンド体系）
- 自動計測データの受信・パース・保存フロー
- デバイスのファームウェアアップデート（OTA）機構
- バックグラウンド BLE 接続の維持と再接続ロジック
- デバイスキャリブレーション（校正）フロー

## 通知・アラートのギャップ

設計ドキュメントでは `expo-notifications` を使用したローカル通知を定義していますが、実装では通知に関するコードが存在しません。`AlertsScreen.tsx` に `notificationsEnabled` と `pairingEnabled` のトグルスイッチが配置されていますが、これらはローカルの React state を切り替えるだけで、実際の通知送信ロジックには接続されていません。

必要な仕様として、通知トリガー条件の定義（どのアラート severity で通知を送るか）、通知チャネルの設定（iOS / Android の通知権限リクエスト）、通知履歴の永続化、プッシュ通知のバックエンド連携（フェーズ 2 以降）、および通知設定のユーザーカスタマイズ UI が挙げられます。

## セキュリティのギャップ

設計ドキュメントでは包括的なセキュリティ対策を定義していますが、実装では多くの項目が未対応です。

| セキュリティ項目 | 設計ドキュメント | 実装状態 |
| :--- | :--- | :--- |
| JWT 認証 | アクセストークン（1 時間）+ リフレッシュトークン（7 日間） | 実装済み（有効期限は `config.ts` で設定） |
| パスワードハッシュ | bcrypt（salt rounds: 12） | 実装済み（サーバー側） |
| RBAC | owner / caretaker / viewer + permissions オブジェクト | `role` 文字列のみ、権限チェックなし |
| SQLCipher 暗号化 | ローカル DB の暗号化 | **未実装**（平文 SQLite） |
| HTTPS 通信 | すべての通信を暗号化 | 設定依存（コード上の強制なし） |
| レート制限 | 1 分間に 100 リクエスト | **未実装** |
| CSRF 対策 | CSRF トークン | **未実装** |
| 個人情報暗号化 | メール、電話番号の暗号化保存 | **未実装** |
| データ削除権 | Right to be forgotten | **未実装**（ユーザー削除 API なし） |
| Secure Storage | トークンの安全な保存 | `AsyncStorage`（平文保存） |

特に、トークンが `AsyncStorage` に平文で保存されている点は、モバイルアプリのセキュリティとして重大な懸念事項です。`expo-secure-store` への移行が強く推奨されます。

## テスト・品質保証のギャップ

設計ドキュメントでは、Jest によるユニットテスト（カバレッジ 80% 以上）、Supertest による統合テスト、Detox による E2E テスト、および React Native Debugger によるパフォーマンステストを定義しています。

**実装の現状は、テストファイルが一切存在しません。** `jest.config.js`、テストユーティリティ、モックファイル、CI/CD パイプライン設定（GitHub Actions）のいずれも確認されませんでした。

テストが存在しないことにより、以下のリスクが生じます。

## オフライン・同期のギャップ

しかし、以下の同期関連の仕様が未定義・未実装です。

- オフライン時のデータキューイング（オフラインで記録したデータをオンライン復帰時に同期）
- コンフリクト解決戦略（同一レコードが複数デバイスから更新された場合の優先ルール）
- 差分同期（全件取得ではなく、変更分のみを同期）
- 同期状態のインジケーター（ユーザーへの同期進捗表示）
- ネットワーク状態の監視と自動切り替え

## 推奨実装ロードマップ

以下に、ギャップの解消に向けた推奨実装順序を示します。各フェーズは、依存関係と優先度を考慮して構成されています。

### フェーズ 1A: MVP 基盤強化（4 週間）

このフェーズでは、MVP リリースに最低限必要な基盤を整備します。

1. **セキュリティ基盤**（1 週間）: `expo-secure-store` への移行、ローカル DB 暗号化（SQLCipher）、RBAC 権限チェックの実装
2. **テスト基盤**（1 週間）: Jest 設定、`health.ts` のユニットテスト、API エンドポイントの統合テスト
3. **API 完備**（1 週間）: ペット更新・削除、計測記録更新・削除、API バージョニング、ページネーション
4. **データモデル修正**（1 週間）: 計測記録への `device_id` 追加、尿検査の `trace` 判定追加、アラートの永続化テーブル追加

### フェーズ 1B: BLE デバイス通信（6 週間）

このフェーズでは、PetSense の中核機能である IoT デバイス連携を実装します。

1. **BLE プロトコル設計**（2 週間）: GATT サービス定義、通信プロトコル仕様書作成、データフォーマット定義
2. **BLE 実装**（2 週間）: `react-native-ble-plx` 統合、スキャン・ペアリング・データ受信の実装
3. **自動計測フロー**（2 週間）: デバイスからの計測データ受信・パース・保存、バックグラウンド接続維持

### フェーズ 1C: 通知・UI 改善（4 週間）

1. **通知システム**（2 週間）: `expo-notifications` 統合、アラートトリガー条件の実装、通知設定 UI
2. **UI 改善**（2 週間）: 計測タイプ別入力、グラフ表示、ペット編集・削除、食事タイプ選択

### フェーズ 2: 同期・拡張機能（8 週間）

1. **オフライン同期**（3 週間）: キューイング、コンフリクト解決、差分同期
2. **拡張データモデル**（2 週間）: 犬種・猫種マスター、医療履歴、画像添付
3. **レポート・エクスポート**（2 週間）: PDF 生成、獣医師共有機能
4. **パフォーマンス最適化**（1 週間）: コード分割、バーチャルスクロール、キャッシング
